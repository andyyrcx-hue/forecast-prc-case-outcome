# 中国类案裁判预测

## 核心原则

将工作分成两个层次：

1. **客观预测层**：完整呈现支持、反对和中性案例，不因我方立场调整相似度、删去不利权威案例或虚构法官倾向。
2. **诉讼表达层**：在事实真实且可核验的前提下，对有利案例强调决定性相同点；对不利案例识别可能改变裁判结果的决定性差异；权威上级法院与承办法官旧案冲突时，优先组织上级法院规则的统一适用理由。

不得把宣传性论证写成客观预测，不得将相关性弱的差异夸大为决定性差异。

## 启动门槛

确认以下信息；只询问会实质改变结果的最少问题：

- 我方诉讼地位与目标判项；
- 案由、程序阶段、受理法院；
- 核心争议焦点和决定性事实；
- 已知承办法官、审判庭或合议庭信息（仅在需要法官轨时）；
- 关键事实的证据状态：已证实、对方争议、材料未体现、待核实。

未给出承办法官时，完成权威案例与事实相似案例检索，并将法官轨标为“材料未体现，未检索”。未给出我方立场时，只做中立预测，不生成偏向性代理表达。

## 隐私与外部检索

元典属于外部检索服务。调用前必须：

1. 将姓名、企业名、案号、合同编号、精确金额、联系方式、流水、证据原文和商业秘密替换为抽象要素；
2. 向用户展示拟发送的脱敏关键词和用途，并取得明确同意；
3. 未经明确授权，不向外部服务发送原始案情、原件全文或可识别信息；
4. 保存检索记录，注明数据源、日期、关键词、脱敏状态、命中范围和局限。

只有在用户明确要求检索特定法官公开裁判并同意发送姓名后，才将法官姓名作为外部查询条件。不得在输出、日志或仓库中保存 API Key、访问令牌或 MCP 配置秘密。

## 工作流

### 1. 建立本案比较基准

将案情拆成法律关系、争议焦点、决定性事实、证据结构、适用规范、程序位置和目标判项。标明信息来源和确定性。读取 [references/scoring-method.md](references/scoring-method.md) 确定评分口径。

先列出 3—7 个一旦改变就可能改变判决的“胜负开关”。后续相似度与差异分析只围绕这些开关展开。

### 2. 生成脱敏检索矩阵

至少准备四组检索式：

- 权威规则：案由 + 核心争点 + 法律概念；
- 事实相似：争点 + 2—4 个决定性事实 + 目标判项；
- 反面案例：相同争点 + 导致不利结果的关键事实；
- 法官轨：法官姓名 + 同案由/同争点；必要时扩展至同审判庭、同法院。

分别扩展同义词、旧法/新法用语和不同案由表述，不用一条长查询替代检索矩阵。

### 3. 探测并调用元典

实际探测可用的元典案例 MCP。只有调用成功才能标注“元典案例”。按 [references/yuandian-guide.md](references/yuandian-guide.md) 执行：

1. 检索权威案例；
2. 扩展普通案例关键词；
3. 补充事实语义检索；
4. 主动检索反面案例；
5. 获取入选案例全文；
6. 经授权后检索法官姓名并核对文书署名。

工具不可用时，停止具体案例与法官倾向结论，输出脱敏检索式并标注 `[待检索]`，不得用模型记忆补造案号、裁判观点或法官履历。

### 4. 初筛与全文核验

列表摘要只用于召回，不用于最终引用。对核心样本文书核验：

- 案号、法院、裁判日期、审级和生效状态；
- 裁判类型，排除不能支持实体观点的程序裁定；
- 法院认定事实、争议焦点、裁判理由、援引规范和判项；
- 是否存在改判、上级法院纠正、旧法失效或事实不完整；
- 法官姓名及其角色是否来自文书署名，而非搜索摘要推断。

### 5. 分开计算三套分数

不得合并成一个不透明总分：

- **事实相似度（S 0—100）**：回答“是不是类案”；
- **权威参照级（A1—A6）**：回答“观点有多强的参照价值”；
- **法官样本强度（J0—J4）**：回答“能否说明承办法官存在相对稳定的公开裁判观察”。

使用 `scripts/score_cases.py` 进行机械计算；法律关系门槛、争点门槛及决定性差异仍由律师判断。

### 6. 建立权威轨与法官轨

**权威轨**按参照价值排序：指导性案例或具有明确统一适用意义的权威案例 → 最高法裁判与典型案例 → 省级高院裁判与典型案例 → 中院 → 基层法院。每一级内部再按事实相似度、地域、时间和全文完整性排序。

**法官轨**按身份可靠性和可比性排序：同一法官同争点 → 同一法官同案由 → 同一合议庭/审判庭 → 同法院。处理重名、调任、不同审判领域和署名角色差异。单案只能说明“观察值”，不能称为稳定倾向。

### 7. 形成预测

分别回答：

- 最可能判项；
- 有利、基准、不利三个情景及触发条件；
- 哪些结果由权威规则驱动，哪些由事实认定、证据缺口或裁量驱动；
- 上级法院观点与承办法官既往做法是否一致；
- 置信度及导致置信度下降的材料缺口。

不输出伪精确胜诉率。只有样本口径一致、数量充分且分母可解释时才给概率区间并说明方法；否则使用“高/中/低”置信度和条件化结论。

### 8. 生成我方论证策略

- 有利类案：只强调与胜负开关有关的相同点，逐项对应本案证据；
- 不利类案：区分决定性事实、证据结构、程序位置或适用法版本，并说明为何足以改变涵摄结果；
- 无法区分的不利权威案例：如实列为核心风险，提出补证、限缩请求、备选论证或和解方案；
- 上级法院有利、承办法官不利：以上级法院统一适用规则为主轴，再用真实事实差异解释旧案；不攻击法官、不隐瞒冲突；
- 上级法院不利、承办法官有利：不得把个别旧案表述为稳定规则，应下调预测并提示改判风险。

### 9. 输出与保存

读取 [references/report-template.md](references/report-template.md)，每次同时生成：

- `类案检索与裁判预测报告.md`：主编辑版；
- `类案检索与裁判预测报告.docx`：正式阅读版；
- `检索记录与关键词.md`：脱敏检索和核验记录。

先完成 Markdown，再从当次最终 Markdown 生成 DOCX。DOCX 必须完整保留结论、案例数据、评分、来源标签、核验状态和风险提示。使用可用的 `documents` Skill 或等价文档能力完成渲染、逐页检查和必要修正后再交付。

如果运行环境不支持 DOCX 生成或渲染，明确标注未完成项和原因，不得把仅交付 Markdown 默认为任务完成。保存位置遵守用户当前工作区规则；未识别工作区时保存到用户明确指定或当前任务可写目录，不创建猜测性的案件目录。

## 质量检查

交付前确认：

- 完整呈现不利权威案例；
- 相似度独立于裁判结果和法院级别；
- 差异点能够连接构成要件、证明责任或判项；
- 法官身份、署名角色和样本数量已经核验；
- 法官观点来自裁判理由，而非仅由判项反推；
- 预测区分已证实事实与待核实事实；
- 明确中国案例的参照边界、检索局限及“需律师核验”。
